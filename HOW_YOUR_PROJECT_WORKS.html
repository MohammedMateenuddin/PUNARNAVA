🔄 How Data Flows Through Your App

Let's trace what happens when a user opens your app, logs in, scans a device, and sees their dashboard:

Step 1: App Starts

Browser loads index.html → main.jsx runs → AuthProvider wraps everything (checks: is someone logged in?) → AppProvider wraps everything (holds scan results) → LanguageProvider wraps everything (English/Hindi) → App.jsx renders → "No user logged in" → Show Login page
What are "Providers"? Think of them as invisible helpers that sit around your entire app and make data available to ANY page. AuthProvider makes login info available everywhere. AppProvider makes scan results available everywhere. Without providers, you'd have to pass data manually from page to page — like passing a note through 10 students instead of announcing it to the class.

Step 2: User Logs In with Google

User clicks "Continue with Google" → AuthContext.jsx calls Firebase's signInWithPopup() → Google popup opens → User picks their account → Google sends back: name, email, photo, unique ID → AuthContext saves this to Firestore database → Sets role = "user" or "recycler" (from toggle) → App.jsx re-renders → Shows UserApp or RecyclerApp
What is Firestore? It's like Google Sheets but for apps. Every user gets a "row" (called a document). Columns include: name, email, points, devicesRecycled, co2Saved, badge. The magic: if you change a value, every screen watching that value updates INSTANTLY — no page refresh needed. This is called "real-time sync."

Step 3: User Scans a Device

User points camera at old phone → Clicks "Capture" → Scanner.jsx captures a photo from the video stream → Calls scanDeviceWithRetry() in geminiScanner.js → Photo converted to base64 (text version of image) → Sent to Google Gemini API with a detailed prompt → Gemini returns JSON: {deviceName, materials, value...} → metalPrices.js fetches LIVE gold/copper prices → Scrap value calculated: weight × price → Results displayed on screen → User clicks "Submit" → Points added to Firestore → Leaderboard updates in real-time

Step 4: Dashboard Shows Impact

User navigates to "My Impact" → Dashboard.jsx reads from AuthContext: currentUser → currentUser.devicesRecycled = 12 → currentUser.co2Saved = 156.3 → currentUser.totalValue = 4230 → AnimatedCounter smoothly counts up to these numbers → Weekly chart distributes devices across days → Tree equivalent = co2Saved ÷ 22 = 7 trees

🧩 Key Concepts You Should Know

1. What is React?

React is a way to build websites using reusable pieces called "components." Instead of writing one giant HTML file, you break your app into small, manageable pieces:

Your App is made of components: App.jsx (the boss) ├── UserSidebar.jsx (the navigation menu) ├── ScrapTicker.jsx (the scrolling price bar at top) ├── Scanner.jsx (the camera + AI page) │ ├── CameraErrorScreen (shown when camera fails) │ ├── MaterialBar (one colored bar per material) │ └── SafetyBadge (the green/yellow/red safety label) ├── Dashboard.jsx (the impact numbers) │ └── AnimatedCounter.jsx (smoothly counts up numbers) ├── Leaderboard.jsx (the rankings) │ └── PodiumItem (top 3 users display) └── Profile.jsx (user info + badges)
Why components matter: If you want to change how the sidebar looks, you edit ONE file (UserSidebar.jsx). Every page using that sidebar updates automatically. Without components, you'd copy-paste the sidebar code into every page — and fixing a bug means editing it everywhere.

2. What is JSX?

JSX is HTML written inside JavaScript. Your .jsx files mix logic and display together:

// This is JSX — it looks like HTML but it's inside JavaScript
function Dashboard() {
  const devicesRecycled = 12;  // ← JavaScript variable

  return (
    <div>                        {/* ← HTML-like syntax */}
      <h1>My Impact</h1>
      <p>Devices: {devicesRecycled}</p>  {/* ← {} inserts JS into HTML */}
    </div>
  );
}

3. What is State? (useState)

State is a variable that, when it changes, automatically re-draws the screen. Normal variables don't do this.

// Normal variable — changing it does NOTHING to the screen
let count = 0;
count = 5;  // Screen still shows 0

// State variable — changing it REDRAWS the screen
const [count, setCount] = useState(0);
setCount(5);  // Screen instantly shows 5!
In your app: scanning, showResults, error, preview in Scanner.jsx are all state variables. When setScanning(true) is called, the scanning animation appears. When setShowResults(true), the results card appears. React handles the visual updates automatically.

4. What is useEffect?

useEffect runs code when the page loads or when specific data changes. Think of it as an automatic trigger.

// "Run this code when the page first loads"
useEffect(() => {
  startCamera();     // Turn on the camera
}, []);              // [] means "only on first load"

// "Run this code whenever scanResult changes"
useEffect(() => {
  fetchDisassemblyVideo(scanResult.name);
}, [scanResult]);    // Runs again if scanResult changes

5. What is Context? (Your App's Shared Brain)

Context solves the problem of "how do I share data between pages?"

WITHOUT Context (painful): App → passes user to → Sidebar → passes user to → Profile App → passes user to → Dashboard App → passes user to → Leaderboard (You'd have to pass "user" manually through every component) WITH Context (your app uses this): AuthContext holds "currentUser" data Any page can grab it instantly with: useAuth() Scanner.jsx: const { currentUser } = useAuth(); Dashboard.jsx: const { currentUser } = useAuth(); Profile.jsx: const { currentUser } = useAuth(); (Everyone reads from the same source — no passing needed)

6. What is an API? (How Your App Talks to External Services)

An API is like a waiter at a restaurant. You (the app) tell the waiter (API) what you want. The waiter goes to the kitchen (Google's servers) and brings back your order (data).

Your app uses 3 APIs: 1. GEMINI API (the AI waiter) You send: a photo + instructions You get back: device name, materials, scrap value, safety info Address: generativelanguage.googleapis.com 2. METALS API (the price waiter) You send: "Give me today's metal prices" You get back: Gold=₹5280/g, Copper=₹452/kg, etc. Address: metals-api.com 3. FIREBASE API (the database waiter) You send: "Save this user's points" or "Get the leaderboard" You get back: confirmation or data Address: firestore.googleapis.com
What is an API Key? It's a password that proves you're allowed to use the API. Without it, the "waiter" refuses to serve you. Your keys are stored in the .env file. The key format looks like: AIzaSy.... If someone steals your key, they can use YOUR free quota — that's why we keep it secret.

7. What is Routing? (How URLs Work in Your App)

When you click "My Impact" in the sidebar, the URL changes to /my-impact. But the page doesn't actually reload — React just swaps out the content. This is called client-side routing.

// In App.jsx — this is your route map:
<Route path="/" element={<Scanner />} />         // localhost:5177/
<Route path="/my-impact" element={<Dashboard />} /> // localhost:5177/my-impact
<Route path="/leaderboard" element={<Leaderboard />} />
<Route path="/profile" element={<Profile />} />

🔬 How the AI Scanner Works (Step by Step)

This is the most impressive part of your app. Here's the complete journey of a single scan:

File: src/utils/geminiScanner.js

The Pipeline:
  1. scanDeviceWithRetry() — the entry point. Enforces a 4.5-second gap between scans (so you don't exceed 15 requests/minute)
  2. scanDevice() — tries every combination of API keys × AI models. If Key 1 + Model 1 fails, it tries Key 1 + Model 2, then Key 2 + Model 1, etc.
  3. scanDeviceWithModel() — the actual API call. Converts your photo to base64, attaches a detailed prompt, sends it to Google, and parses the response
  4. extractJSON() — cleans up the AI's response (sometimes Gemini adds markdown or extra text around the JSON)
  5. mapToUI() — takes the raw AI data and formats it for your screen. Fetches live metal prices and calculates real scrap values
  6. If EVERYTHING fails → getFallbackResult() picks from a built-in database of 6 device types

The Prompt (What You Tell the AI)

The quality of your AI results depends entirely on this prompt. Here's the key part:

"You are an e-waste SCRAP VALUE analyst.
Analyze this image and identify the electronic device.

IMPORTANT RULES:
- Scroll wheel + buttons = MOUSE (not laptop)
- Glass slab with touchscreen = SMARTPHONE
- Clamshell with keyboard = LAPTOP

PRICING RULE:
- Give SCRAP value, NOT retail price
- Gold: ₹5200/g, Copper: ₹450/kg, Aluminium: ₹120/kg

Return ONLY raw JSON: {deviceName, materials, value...}"
To improve accuracy: Edit the prompt in geminiScanner.js (line ~263). The more specific your instructions, the better Gemini performs. You can add new device types, adjust pricing rules, or add new output fields.

⚠️ Things to Fix Before Going Live

  • Security rules: Your Firestore database currently allows anyone to read/write. Add security rules in Firebase Console → Firestore → Rules.
  • API key protection: Move Gemini API calls to a backend server so your key isn't exposed in browser code.
  • Real payment gateway: Replace the mock payment with Razorpay/Stripe.
  • Error tracking: Add Sentry or LogRocket to catch errors from real users.
  • Testing: Add automated tests so changes don't break existing features.
